# Green VPN — Project Overview

## What this is

A minimal Android VPN client that routes all device traffic through an xray-core proxy.
No config UI — the server config is bundled in `assets/config.json`.
Currently wired for VLESS + REALITY + xtls-rprx-vision, but supports any protocol
xray-core speaks (VMess, Trojan, Shadowsocks, etc.) by changing the config.

## Components

### `go/` — xray-core Go wrapper

**Files:** `libgreen.go`, `imports.go`, `go.mod`

We own this code. It wraps xray-core via gomobile and exposes three things to Kotlin:

- `Libgreen.setProtector(Protector)` — registers a socket protection callback
- `Libgreen.start(configJSON)` — starts xray-core with a JSON config
- `Libgreen.stop()` — stops xray-core

The `Protector` interface lets `GreenVpnService` call Android's `VpnService.protect(fd)`
on every socket xray opens. Without this, xray's outbound traffic would re-enter tun0
and loop infinitely.

Socket protection is wired via `internet.RegisterDialerController` from xray-core's
transport layer — it fires a callback for every outbound connection before it's established.

`imports.go` exists purely for blank imports. xray-core uses Go `init()` functions to
register protocol and transport handlers at startup. Without importing those packages,
the JSON config loader won't know what VLESS, REALITY, etc. are.

**Why we own this instead of using a prebuilt:**
The main prebuilt option (`AndroidLibXrayLite`) removed `DialerController` support in
recent versions, which meant no socket protection and a split-route workaround. Owning
the Go code gives us control over the API, dependency versions, and the ability to add
socket protection properly.

**Build:** `./gradlew buildGoLib` — Gradle runs `gomobile bind` and outputs `libgreen.aar`
into `app/libs/`. The task is input-tracked: it only reruns when Go sources change.

### `app/libs/libgreen.aar` — generated artifact

Output of `gomobile bind`. Contains:

- `libgojni.so` (arm64-v8a only) — the Go runtime + xray-core compiled for Android
- `classes.jar` — Java stubs generated by gomobile for the Kotlin API

~20MB. Not checked into git — rebuilt from `go/` sources.

---

### `app/src/main/jniLibs/arm64-v8a/libhev-socks5-tunnel.so`

**What it does:** Reads raw IP packets off the tun0 file descriptor, runs a userspace
TCP/IP stack, and forwards connections as SOCKS5 requests to xray on `127.0.0.1:10808`.

**Why it's needed:** Android's VPN interface gives you raw IP packets. xray speaks SOCKS5.
Something has to translate between the two — that's hev-socks5-tunnel (tun2socks).

**Why hev specifically:** It's what v2rayNG ships, it's a single `.so` with a dead-simple
JNI interface (`TProxyStartService(configPath, fd)` / `TProxyStopService()`), and the
config is just a small YAML file we write at runtime.

**Why it lives at `com.v2ray.ang.service.TProxyService`:** The `.so` has that class path
hardcoded in its `JNI_OnLoad` `RegisterNatives` call. Our Kotlin class must live at that
exact package for the JNI binding to succeed.

**Source:** Extracted from v2rayNG APK (GPL-3). Not checked into git — documented in
`docs/poc-plan.md`. For production, build from https://github.com/heiher/hev-socks5-tunnel.

### `GreenVpnService` — the glue

Subclasses Android's `VpnService`. Owns the lifecycle of all three native components.

Startup sequence:

1. Post foreground notification (Android requires this for VPN services)
2. Register socket protector with xray via `Libgreen.setProtector()`
3. Build the TUN interface — `VpnService.Builder` → `tun0`
4. Start xray: `Libgreen.start(config)` — listens on `127.0.0.1:10808` SOCKS5
5. Write hev YAML config to `filesDir/tun.yaml`
6. Start hev: `TProxyService.TProxyStartService(yamlPath, tun0Fd)`

Shutdown reverses the order: stop hev → stop xray → close TUN fd.

Routing: `addRoute("0.0.0.0/0")` + `addRoute("::/0")` — all traffic goes through tun0.
This is safe because step 2 protects xray's own sockets before the tunnel opens.

---

### `VpnViewModel` + `VpnState`

`VpnViewModel` handles the one awkward Android detail: `VpnService.prepare()` may show
a system dialog asking the user to grant VPN permission. The result comes back via
`ActivityResult`, so the ViewModel bridges that into a simple `connect()` call.

`VpnState` is a global `MutableStateFlow<VpnStatus>` — the service writes to it,
the Compose UI reads from it. Simple for a PoC; for a larger app you'd use a bound
service or broadcast receiver instead.

---

### `assets/config.json`

xray-core JSON config, read at connect time. Must contain:

- A SOCKS5 inbound on `127.0.0.1:10808` (where hev sends traffic)
- At least one outbound (your proxy server)

The routing section is optional — without it, all traffic goes to the first outbound.

## Key design decisions

**Why xray-core + hev instead of sing-box?**
sing-box has built-in TUN support and would eliminate hev entirely, but the user's
server is already configured for xray protocols. xray-core speaks that natively.
sing-box is the better long-term choice if starting fresh.

**Why arm64-v8a only?**
Covers ~95% of phones sold in the last 4 years. Keeps the APK small and the build simple.
Adding `armeabi-v7a` (32-bit) is straightforward if needed.

**Why not use Android's split-tunnel (`addAllowedApplication`) instead of socket protection?**
Split-tunnel is per-app. Socket protection is per-socket. We need the VPN to cover all
apps, and we need xray's own traffic to bypass it — those are orthogonal axes.

**Why `198.18.0.1` as the TUN address?**
It's in the IANA-reserved benchmarking range (198.18.0.0/15, RFC 2544) — guaranteed to
never be a real internet address, so it won't conflict with anything on a normal network.
